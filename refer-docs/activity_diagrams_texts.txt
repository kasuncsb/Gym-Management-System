UC-01 — Scan QR Code (Activity Diagram)
---------------------------------------
Primary actors: Member (or Staff scanning on member's behalf); QR scanner device; Gym System; Authentication service
Start: Member approaches entrance with QR (displayed on phone or printed) and opens scanner or presents QR to kiosk.

Main flow (step-by-step):
  1. Scanner attempts to read the QR code (reads memberID/session token).
  2. System receives scanned QR payload and looks up member profile from the Membership database.
  3. System invokes Validate Subscription (UC-02) with memberID.
  4. System invokes Authenticate User (UC-03) to confirm identity/session token validity.
  5. If authentication and subscription validation succeed, system calls door unlocking endpoint and unlocks entrance gate.
  6. System calls Record Timestamp (UC-04) to log IN (or OUT) event with device/gate metadata.
  7. System returns success acknowledgement to the scanner UI (door opened and timestamp logged).
  8. End.

Decision nodes and branches:
  - Was QR readable? -> Yes: continue. No: show 'retry' and allow manual memberID input or alternate authentication.
  - Is subscription valid? -> Yes: continue. No: show 'subscription invalid' message and offer Buy Subscription (UC-05).
  - Is authentication valid? -> Yes: unlock door. No: show authentication error and deny entry.

Extensions / Alternate flows:
  - QR unreadable: scanner prompts retry or manual entry of member ID.
  - Subscription invalid or expired: system prompts to Buy/Renew subscription or denies access.
  - Authentication failure: display error and deny entry.
  - Timestamp logging unavailable (DB offline): store event locally with a retry/sync queue and notify admin.

Data outputs / postconditions:
  - Door unlock command (on success).
  - Persistent IN/OUT timestamp record (memberID, time, gateID, deviceID) or queued local store if offline.
  - User-visible success or failure message.

Notes:
  - Include protections such as signed/one-time QR tokens, rate-limiting to prevent replay attacks, low-latency requirement at entry gate.
  - Activity diagram shows inclusion of UC-02, UC-03, and UC-04 as sub-activities.

UC-02 — Validate Subscription (Activity Diagram)
------------------------------------------------
Primary actors: Gym System (subsystem); Billing/subscriptions service; Membership database; Payment gateway (for renewal flows)
Start: A service call arrives with the memberID and the requested entitlement check.

Main flow (step-by-step):
  1. Receive validation request with memberID and requested entitlements (e.g., facility access, class booking).
  2. Lookup member subscription record(s) in the Membership/Billing database.
  3. If multiple subscriptions exist, resolve priority and compute effective entitlements.
  4. Check subscription expiry, access hours, facility restrictions and special entitlements.
  5. Return validation result: valid (with entitlements and expiry) or invalid with reason code.

Decision nodes and branches:
  - Does member have any active subscription? -> Yes: compute entitlements and return success. No: return invalid/expired.
  - Multiple active subscriptions? -> Resolve priority (e.g., premium overrides basic) and return effective entitlement.

Extensions / Alternate flows:
  - Subscriptions service/database unavailable -> return error code indicating service down and suggest offline fallback or retry.
  - Ambiguous subscription records -> flag for admin/manual resolution.

Data outputs / postconditions:
  - Validation response with status (valid/invalid), expiry date, and entitlement object (e.g., allowed hours, booked classes).

Notes:
  - This UC is frequently called as a subroutine by access, booking, and payment flows.

UC-03 — Authenticate User (in/out) (Activity Diagram)
-----------------------------------------------------
Primary actors: Member; Staff (manual auth); Authentication service; Identity Provider (IDP); Device (mobile/kiosk)
Start: Member presents QR or credentials for authentication.

Main flow (step-by-step):
  1. Receive authentication request with credentials or QR/session token.
  2. Verify token signature, expiry and membership association.
  3. Optionally call external IDP or federated auth provider for additional verification.
  4. If valid, mark the member as authenticated for the current session and return success token/acknowledgement.
  5. If invalid, return authentication failure and error message.

Decision nodes and branches:
  - Is the token valid and unexpired? -> Yes: authenticated. No: authentication denied.
  - Is two-factor or staff override required? -> Proceed with additional verification or staff manual override.

Extensions / Alternate flows:
  - Invalid QR or expired token -> deny entry and instruct user to refresh QR or contact support.
  - Auth service down -> deny authentication and use fallback (manual staff check-in) if permitted.

Data outputs / postconditions:
  - Authentication result. On success, issue session confirmation used by door controller and timestamp recording.

Notes:
  - Authentication is tightly coupled with subscription validation to avoid unauthorized access.

UC-04 — Record Timestamp (in/out) (Activity Diagram)
----------------------------------------------------
Primary actors: Gym System backend; Time-series DB; Member profile service; Door/gate device
Start: Authentication success or manual IN/OUT trigger occurs.

Main flow (step-by-step):
  1. Receive IN/OUT event with memberID, event type (IN or OUT), deviceID, gateID and timestamp.
  2. Validate event ordering to prevent duplicate IN or OUT (e.g., if last event is IN and new event is IN, flag duplicate).
  3. Persist timestamp entry to time-series or event log database with metadata.
  4. Return an acknowledgement to calling service.
  5. If persistence fails, queue write for retry and notify admin.

Decision nodes and branches:
  - Is event ordering valid (not duplicate)? -> Yes: persist. No: flag and notify staff for review.
  - DB write success? -> Yes: ack. No: queue locally and alert admin.

Extensions / Alternate flows:
  - Duplicate IN detected -> flag for staff review and optionally deny second IN.
  - DB write fails / offline -> store locally and sync when service restored.

Data outputs / postconditions:
  - Persistent timestamp record used for attendance reports, analytics and progress tracking.

Notes:
  - Timestamp entries must be tamper-resistant and contain device/gate metadata for auditability.

UC-05 — Buy Subscription (Activity Diagram)
-------------------------------------------
Primary actors: Member; Payment gateway; Billing service; Notification service; Validate subscription (UC-02)
Start: Member selects a plan to purchase or renew from UI (app/website).

Main flow (step-by-step):
  1. Show available plans, terms and price; member selects plan and payment method.
  2. Create a payment order and redirect to payment gateway or process in-app payment.
  3. Wait for payment gateway callback confirming success or failure.
  4. On payment success, billing service activates/updates subscription record for the member.
  5. Call Validate Subscription (UC-02) to confirm updated entitlement visibility across system.
  6. Send confirmation notification/email to member.
  7. End.

Decision nodes and branches:
  - Payment success? -> Yes: activate subscription and notify. No: display error and allow retry or alternative payment.
  - Is activation immediate or scheduled? -> If scheduled, queue activation and notify member with expected time.

Extensions / Alternate flows:
  - Payment declined -> rollback any partial changes and notify the member with instructions to retry.
  - Billing service failure on activation -> schedule retry and inform member of delay.

Data outputs / postconditions:
  - Updated subscription record; confirmation notifications; billing receipts.

Notes:
  - Handle concurrency and idempotency for multiple payment callbacks; ensure secure handling of payment data.

UC-06 — Make an Appointment (Activity Diagram)
----------------------------------------------
Primary actors: Member; Instructor/Trainer; Scheduling system; Notification service; Validate subscription (UC-02)
Start: Member opens scheduling UI and chooses a trainer and time slot.

Main flow (step-by-step):
  1. Member browses available slots and selects preferred trainer and time.
  2. System verifies member entitlement via Validate Subscription (UC-02).
  3. System checks trainer availability and concurrently reserves the slot (optimistic lock or transactional reservation).
  4. On successful reservation, appointment is saved to calendar and notifications sent to member and trainer.
  5. Optionally create calendar entries and reminder notifications.
  6. End.

Decision nodes and branches:
  - Is member entitled to book this slot? -> Yes: proceed. No: suggest upgrade (UC-05).
  - Was slot still available at reservation time? -> Yes: confirm. No: present alternatives.

Extensions / Alternate flows:
  - Concurrent booking conflict -> offer next available slots or place on waitlist.
  - Member cancels -> apply cancellation policy and notify trainer; free up the slot.

Data outputs / postconditions:
  - Appointment record in scheduling system; notifications; calendar entries.

Notes:
  - Support cancellation policies and waitlists; ensure notifications to avoid no-shows.

UC-07 — Mark Workout Done (Activity Diagram)
--------------------------------------------
Primary actors: Member; Workout scheduler; Progress tracking module; Notification service
Start: Member opens an active workout session or scheduled routine in app.

Main flow (step-by-step):
  1. Member presses 'Done' on the workout session or system auto-detects completion.
  2. System verifies session identity and marks workout as completed for that member and session.
  3. Record timestamp for completion (through UC-04 if applicable).
  4. Progress metrics are recalculated (consistency, totals, personal records) and stored.
  5. Update member's progress dashboard and send optional congratulatory notification.
  6. End.

Decision nodes and branches:
  - Was workout legitimately complete? -> Yes: mark complete. No: prompt to confirm or cancel.
  - Is device offline? -> Persist locally and sync later.

Extensions / Alternate flows:
  - Failure to update progress -> show retry option and queue update for background sync.
  - Member attempts to mark prematurely -> prompt confirmation.

Data outputs / postconditions:
  - Updated progress record, recalculated metrics, and completion timestamp.

Notes:
  - Support offline persistence for mobile clients and eventual consistency across analytics.

UC-08 — View / Track Progress (Activity Diagram)
------------------------------------------------
Primary actors: Member; Progress analytics engine; Data storage; Mark workout done (UC-07)
Start: Member navigates to the Progress or Dashboard screen.

Main flow (step-by-step):
  1. System requests member's historical activity and workout completion data from storage.
  2. Analytics engine aggregates activity history, measurements and computes trends (e.g., consistency, volume).
  3. Compose visualizations (graphs, summaries) and suggestions based on analytics output.
  4. Display dashboard to member with drill-downs and suggested next steps.
  5. End.

Decision nodes and branches:
  - Is there sufficient data for analytics? -> Yes: show trends. No: show tips to get started or request permission for wearable integration.

Extensions / Alternate flows:
  - Missing or partial data -> display partial results with explanatory message.
  - Permission missing (e.g., wearable integration not authorized) -> show steps to authorize.

Data outputs / postconditions:
  - Visualized analytics, trend lines, suggestions, and downloadable progress reports.

Notes:
  - Analytics may be batched or computed on-demand depending on system load.

UC-09 — Enter / Leave Gym (Activity Diagram)
--------------------------------------------
Primary actors: Staff (primary); Door control system; Scan QR Code (UC-01); Record timestamp (UC-04)
Start: Staff member attempts to enter or leave facility (using QR or presence detection).

Main flow (step-by-step):
  1. Access request is detected (QR scan or Wi-Fi presence detection).
  2. System validates and authenticates the staff via UC-02 and UC-03 as required.
  3. Door unlocks on success and system records timestamp via UC-04.
  4. System logs access event and returns acknowledgement.
  5. End.

Decision nodes and branches:
  - Is door hardware functioning and network available? -> Yes: proceed. No: deny or use fallback.
  - Is staff authorized? -> Yes: unlock. No: deny entry and alert security/admin.

Extensions / Alternate flows:
  - Network or door hardware failure -> deny entry or require manual override.
  - Access denied due to insufficient authorization -> alert admin.

Data outputs / postconditions:
  - Access log entries; door control command; alerts if failure.

Notes:
  - Staff flows may allow manual overrides or separate staff cards with elevated privileges.

UC-10 — Change / Modify Workout Schedule (Activity Diagram)
-----------------------------------------------------------
Primary actors: Member; Instructor; Workout scheduler; Notification service; Progress module
Start: User (member or instructor) selects a workout schedule entry to edit.

Main flow (step-by-step):
  1. Open existing workout plan entry in scheduler UI.
  2. Edit exercises, times or participants as required.
  3. System verifies constraints (trainer availability, member entitlements) before applying changes.
  4. Save changes to schedule and notify affected participants (members/trainers).
  5. If conflict arises, present alternatives or request approval flow.
  6. End.

Decision nodes and branches:
  - Does the change conflict with existing appointments or trainer availability? -> Yes: present alternatives or queue for approval. No: commit change.
  - Does change affect multiple participants? -> Notify all affected parties.

Extensions / Alternate flows:
  - Conflict detected -> reject or require admin/trainer approval.
  - Save fails -> notify user and provide retry option.

Data outputs / postconditions:
  - Updated schedule entries, notifications and updated progress context if needed.

Notes:
  - Maintain audit trail for schedule changes for accountability and reporting.

UC-11 — View Insights (Activity Diagram)
----------------------------------------
Primary actors: Manager; Instructor; Analytics engine; Data warehouse; Notifications
Start: Manager or instructor requests aggregated insights dashboard.

Main flow (step-by-step):
  1. User opens Insights page or a scheduled job triggers proactive insights.
  2. System queries aggregated attendance, revenue, utilization and performance data from the data warehouse.
  3. Analytics/AI engine computes suggested actions, trends and anomalies.
  4. System composes the insights dashboard and sends notifications if configured.
  5. End.

Decision nodes and branches:
  - Is there sufficient aggregated data for requested insight? -> Yes: present insights. No: indicate insufficiency and suggest methods to collect more data.

Extensions / Alternate flows:
  - Computation failure or timeout -> return partial results and show an explanatory warning.
  - Insufficient data for a specific insight -> present fewer widgets or suggest data collection steps.

Data outputs / postconditions:
  - Insights dashboard, recommended actions, and exportable reports.

Notes:
  - This flow may support scheduled background analytics and real-time on-demand queries.

UC-12 — Access Detailed Reports (Activity Diagram)
--------------------------------------------------
Primary actors: Manager/Admin; Reporting subsystem; Database; Export service (PDF/CSV); AuthN/Z service
Start: Manager selects report type, date range and filters in reporting UI.

Main flow (step-by-step):
  1. Report request received with parameters and user credentials.
  2. System validates the user's permissions and roles.
  3. System retrieves and aggregates raw data from the database for requested range.
  4. Report compiled and formatted; export options (PDF/CSV) prepared.
  5. Provide download link or stream of the generated report and log report access.
  6. End.

Decision nodes and branches:
  - Does user have permission to access requested report? -> Yes: generate. No: deny request.
  - Is data volume too large for on-demand generation? -> Schedule a background job and notify user when ready.

Extensions / Alternate flows:
  - Report generation error -> log and inform user with troubleshooting steps.
  - Export service failure -> retry and notify admin if persistent.

Data outputs / postconditions:
  - Generated report file(s), access logs, and notifications.

Notes:
  - Provide paginated previews or sample to avoid generating extremely large ad-hoc reports unnecessarily.

UC-13 — Manage the Entire System (Activity Diagram)
---------------------------------------------------
Primary actors: System Administrator; All subsystem services (billing, auth, scheduler, door controllers)
Start: Administrator logs into Admin console and selects management function.

Main flow (step-by-step):
  1. Admin selects a management action (create plan, change pricing, add location, manage roles).
  2. System validates admin privileges and input parameters.
  3. Perform changes on targeted subsystems (update pricing config, provision hardware endpoints, update RBAC).
  4. Persist configuration changes and inform dependent services (webhooks or service bus).
  5. Log the admin action for audit and optionally notify stakeholders.
  6. End.

Decision nodes and branches:
  - Is the admin authorized for this operation? -> Yes: proceed. No: deny and log attempt.
  - Does the change require coordinated updates across services? -> Yes: perform transactional updates or rollbacks on partial failure.

Extensions / Alternate flows:
  - Partial update failure -> rollback if possible and alert admin with diagnostic info.
  - Invalid input -> reject and show validation errors.

Data outputs / postconditions:
  - Updated system configuration, audit logs and notifications to affected services/users.

Notes:
  - Ensure auditability, role-based access controls and safe rollbacks for critical system updates.
