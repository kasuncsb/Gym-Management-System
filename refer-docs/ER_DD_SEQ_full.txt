--- ENTITY-RELATIONSHIP DIAGRAM (textual description) ---
The ER diagram represents the database structure for the gym management system. Main entities and relationships (textual):
- MEMBER (member_id PK) — has SUBSCRIPTION(s), ATTENDANCE records, PAYMENT records, WORKOUT_SCHEDULE(s), WORKOUT_SESSION(s), PROGRESS entries, DOOR_ACCESS attempts, APPOINTMENT(s).
- TRAINER (trainer_id PK) — can have APPOINTMENT(s), WORKOUT_SESSION(s) with members, referenced by DOOR_ACCESS and schedule features.
- STAFF (staff_id PK) — administers the system, modifies SYSTEM_CONFIG, may record ATTENDANCE (for staff), and do DOOR_ACCESS actions.
- SUBSCRIPTION_PLAN (plan_id PK) — referenced by SUBSCRIPTION.
- SUBSCRIPTION (subscription_id PK) — references MEMBER and SUBSCRIPTION_PLAN.
- ATTENDANCE (attendance_id PK) — references MEMBER.
- DOOR_ACCESS (access_id PK) — references MEMBER or TRAINER (nullable).
- APPOINTMENT (appointment_id PK) — references MEMBER and TRAINER.
- WORKOUT_SCHEDULE (schedule_id PK) — references MEMBER.
- WORKOUT_SESSION (session_id PK) — references MEMBER and can be linked to SESSION_EQUIPMENT rows.
- SESSION_EQUIPMENT (composite PK session_id + equipment_id) — links WORKOUT_SESSION and EQUIPMENT.
- EQUIPMENT (equipment_id PK) — referenced in SESSION_EQUIPMENT and WORKOUT_SESSION.equipments_used (JSON).
- PROGRESS (progress_id PK) — references MEMBER (and optionally session_id).
- PAYMENT (payment_id PK) — references MEMBER and SUBSCRIPTION.
- SYSTEM_CONFIG (config_id PK) — stores key-value system settings.

Relationships are typical: one-to-many from MEMBER to most activity tables, many-to-many between sessions and equipment resolved by SESSION_EQUIPMENT.

--- COMPLETE DATA DICTIONARY (tables and columns) ---

MEMBER Table
Description: Stores all gym member information and authentication details
Columns:
- member_id VARCHAR(20) PRIMARY KEY — Unique member identifier (example: MEM001)
- name VARCHAR(100) NOT NULL — Full name (example: John Smith)
- email VARCHAR(100) UNIQUE, NOT NULL — Contact email (example: john.smith@email.com)
- phone VARCHAR(15) NULL — Phone number (example: +94123456789)
- date_of_birth DATE NULL — Birth date (1990-05-15)
- join_date DATE NOT NULL — Membership start date (2024-01-15)
- emergency_contact VARCHAR(100) NULL — Emergency contact (example: Jane Smith - 0771234567)
- password_hash VARCHAR(255) NULL — Hashed password (bcrypt hash)
- qr_code_token VARCHAR(100) UNIQUE — QR code token (example: qr_abc123def456)
- status ENUM('active','inactive','suspended') DEFAULT 'active' — Status (example: active)
- created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP — Creation timestamp (2024-01-15 09:30:00)

TRAINER Table
Description: Stores trainer information and authentication details
Columns:
- trainer_id VARCHAR(10) PRIMARY KEY — Unique trainer identifier (TRN001)
- name VARCHAR(100) NOT NULL — Full name (Sarah Johnson)
- specialization VARCHAR(100) NULL — Specialization (Weightlifting, Yoga)
- email VARCHAR(100) UNIQUE, NOT NULL — Contact email (sarah.j@email.com)
- phone VARCHAR(15) NULL — Phone number (+94123456788)
- availability JSON NULL — Availability schedule (example: {"mon": ["09:00-17:00"]})
- hourly_rate DECIMAL(8,2) NULL — Hourly rate (2500.00)
- status ENUM('active','inactive') DEFAULT 'active'
- password_hash VARCHAR(255) NULL
- qr_hash VARCHAR(100) UNIQUE — QR hash for access (qr_trainer_xyz789)

STAFF Table
Description: Stores staff members information and roles

- staff_id VARCHAR(10) PRIMARY KEY — Unique staff identifier (STF001)
- name VARCHAR(100) NOT NULL — Full name (Mike Davis)

- email VARCHAR(100) UNIQUE, NOT NULL — Contact email (mike.d@email.com)
- phone VARCHAR(15) NULL — Phone number (+94123456787)
- permissions JSON NULL — Permissions (example: {"can_manage_members": true})

--- SEQUENCE DIAGRAMS (textual transcriptions and reconstructed flows) ---
Note: The original file contains sequence diagrams as images. Below are textual reconstructions of the message flows and interactions for each use case, inferred from context and the data model. They are intended to capture the same high-level interaction steps that the diagrams represent.

UC-01: Scan QR Code - Sequence Diagram
Actors: Member (mobile app or printed QR), QR Scanner (gate), Access Service, Subscription Service, Attendance Service, Door Controller
Flow:
1. Member presents QR code to Scanner (or opens QR in app).
2. Scanner reads QR token and sends a "validateQR(token, gateId, deviceId)" request to the Access Service.
3. Access Service checks QR token -> looks up MEMBER by qr_code_token or qr_hash.
4. Access Service calls Subscription Service to validate subscription status for member (start_date, end_date, status).
5. If subscription valid and status 'active', Access Service records attendance by calling Attendance Service (event_type=IN/OUT, timestamp, gate_id, device_id).
6. Access Service records a DOOR_ACCESS entry (access_id) with status 'granted' and reason NULL.
7. Access Service sends command to Door Controller to unlock door (or return 'grant' to gate hardware).
8. Scanner/Front-end shows "Access Granted" to member; otherwise show "Access Denied" with reason.
9. Optionally, system notifies member via app or logs for analytics.

UC-02: Validate Subscription - Sequence Diagram
Actors: Front-end (App / Scanner / Web), Subscription Service, SUBSCRIPTION table, PAYMENT table
Flow:
1. Front-end requests "validateSubscription(memberId or token)".
2. Subscription Service retrieves current subscription record(s) for member from SUBSCRIPTION table.
3. Checks dates (start_date <= today <= end_date) and status ENUM.
4. If payment_status is 'paid' and status 'active', return VALID.
5. If pending payment or expired, Subscription Service may check PAYMENT records to see if payment confirmation exists or return INVALID/EXPIRED.
6. Service returns validation result and optional next steps (renew, pay).

UC-03: Authenticate User - Sequence Diagram
Actors: User (mobile/web/staff), Auth Service, MEMBER/TRAINER/STAFF table, Session Service (tokens)
Flow:
1. User provides credentials (email+password) or QR token or staff credentials.
2. Auth Service checks corresponding table to find password_hash or qr_hash/qr_code_token.
3. Auth Service verifies password (bcrypt) or validates QR token.
4. On success, Auth Service issues session token (JWT or server session) and returns user profile and roles/permissions.
5. Auth Service logs authentication attempt and returns relevant session expiry.

UC-04: Record Timestamp - Sequence Diagram
Actors: Scanner/Device, Attendance Service, ATTENDANCE table, MEMBER table
Flow:
1. Device sends event 'checkIn(memberId or token, gateId, deviceId)'.
2. Attendance Service resolves member (if token used).
3. Attendance Service creates ATTENDANCE row with event_type IN/OUT and timestamp.
4. Optionally update last_seen or analytics, update member's current presence state.
5. Response back to device confirming success.

UC-05: Buy Subscription - Sequence Diagram
Actors: Member (app/web), Subscription Service, SUBSCRIPTION_PLAN table, Payment Gateway, PAYMENT table
Flow:
1. Member selects a SUBSCRIPTION_PLAN and initiates purchase in app.
2. App calls Subscription Service to create a new SUBSCRIPTION record in 'pending' state.
3. Subscription Service creates PAYMENT record with status 'pending' and redirects the user to Payment Gateway (or uses API).
4. User completes payment on Payment Gateway; Gateway returns success/failure via callback/webhook.
5. On payment success webhook, Subscription Service marks PAYMENT.status='success', updates SUBSCRIPTION.status='active', sets payment_status='paid', sets start_date/end_date based on plan.duration_days.
6. Service notifies member and possibly issues QR update or sends confirmation email/receipt.

UC-06: Make an Appointment - Sequence Diagram
Actors: Member, Front-end, Appointment Service, TRAINER table, Notification Service
Flow:
1. Member chooses trainer and desired date/time.
2. Front-end asks Appointment Service to "createAppointment(memberId, trainerId, date, startTime, endTime)".
3. Appointment Service checks trainer availability (TRAINER.availability and existing appointments).
4. If available, Appointment Service creates APPOINTMENT record with status 'scheduled' and notifies trainer via Notification Service.
5. On trainer confirmation/completion, status moves to 'completed' or 'cancelled/no-show'.

UC-07: Mark Workout Done - Sequence Diagram
Actors: Member or Trainer, Workout Session Service, WORKOUT_SESSION and WORKOUT_SCHEDULE, SESSION_EQUIPMENT, PROGRESS
Flow:
1. Member/trainer selects scheduled workout and marks it as started/completed via app or staff terminal.
2. Workout Session Service creates or updates WORKOUT_SESSION record (start_time, end_time).
3. Equipment used is recorded by adding JSON to equipments_used or creating SESSION_EQUIPMENT entries.
4. On completion, optionally update PROGRESS with metrics (calories, weight lifted) and notify member.

UC-08: View/Track Progress - Sequence Diagram
Actors: Member, Progress Service, PROGRESS table, WORKOUT_SESSION table, Reporting/Aggregation Service
Flow:
1. Member requests progress dashboard.
2. Progress Service queries PROGRESS and WORKOUT_SESSION tables for member.
3. Aggregation calculates trends (weight over time, calories burned, session counts).
4. Service returns structured data for charts and details.

UC-09: Enter/Leave Gym (Staff) - Sequence Diagram
Actors: Staff, Door Scanner/Staff Terminal, Auth Service, DOOR_ACCESS table, ATTENDANCE
Flow:
1. Staff authenticates via credentials or staff QR.
2. Scanner sends staff identity to Auth Service for validation and role check.
3. On success, DOOR_ACCESS record is created with trainer_id or staff_id as appropriate; ATTENDANCE may also be recorded.
4. Door opens and the system logs entry/exit.

UC-10: Change/Modify Workout Schedule - Sequence Diagram
Actors: Member or Staff, Schedule Service, WORKOUT_SCHEDULE table, Notification Service
Flow:
1. User requests a schedule change (reschedule/cancel).
2. Schedule Service checks for conflicts and trainer availability.
3. Service updates WORKOUT_SCHEDULE record (scheduled_date/time/status).
4. Notify member and trainer of the change.

UC-11: View Insights - Sequence Diagram
Actors: Admin, Insights Service, Multiple tables (PAYMENT, ATTENDANCE, WORKOUT_SESSION, PROGRESS), Reporting Service
Flow:
1. Admin requests insights.
2. Insights Service queries and aggregates data (revenue, active members, attendance rates, equipment usage).
3. Service returns dashboards or CSV reports.

UC-12: Access Detailed Reports - Sequence Diagram
Actors: Admin, Reporting Service, DB, Export Service
Flow:
1. Admin requests a detailed report (date range, type).
2. Reporting Service pulls data from relevant tables and formats report (CSV/PDF).
3. Service provides file download and/or schedules periodic reports.

UC-13: Manage Entire System - Sequence Diagram
Actors: Admin (Staff with 'admin' role), Admin Service, All CRUD services, SYSTEM_CONFIG
Flow:
1. Admin performs actions (create/update/delete members, trainers, plans, equipment, config).
2. System validates permissions (Staff.permissions).
3. Admin Service performs requested DB operations and logs changes (SYSTEM_CONFIG.last_modified, modified_by).
4. Changes propagate to dependent services (access rules, subscription validations).
